# Бизнес-процессы системы / Business Processes

## Актуализация от 17.03.2026

- **Единая система уведомлений**: все сообщения об ошибках, предупреждения, подтверждения действий отображаются в одном стиле — по центру экрана с blur-оверлеем и иконкой-индикатором.
- **ФИО оператора ШИГ/КУГ**: обязательное поле для всех действий в форме GMC (сохранение оценки, любого решения), валидация блокирует кнопки до ввода, используется вместо обобщённого названия роли в логах.
- В предупреждении о неполных данных бенефициара список отсутствующих полей отображается локализованными лейблами (TJ/RU).
- В форме договора добавлена нижняя кнопка `Бекор кардан / Отмена` рядом с действиями сохранения и печати.

> **Двуязычность**: Все тексты интерфейса (кнопки, бейджи, оповещения, заголовки, поля форм, критерии оценки, документы, CSV-экспорт) являются двуязычными — таджикский текст отображается крупнее, русский — мельче. Ни одна надпись не существует на одном языке.

## Актуализация от 13.03.2026

- При 3-м неодобрении заявка переводится в `postponed` и блокируется на 3 месяца.
- Автоматической реактивации после 3 месяцев нет.
- После истечения срока заявка получает признак `unlock-ready` (готова к разблокировке), но остается в `postponed` до ручного действия Фасилитатора.
- Разблокировка выполняется только вручную кнопкой `Разблокировать` (из карточки/таблицы или из ленты уведомлений).
- После ручной разблокировки заявка переходит в `fac_revision`, `reactivated = true`, `revisionCount = 0`.
- Документный пакет бизнес-плана: версионируется только Word-файл (`V1`, `V2`, ...).
- PDF и фото-комплект являются фиксированными вложениями базовой версии.
- При первичной отправке обязательны: Word + PDF + ровно 4 фото.
- В карточках/таблицах отображается `Current Word Version: Vn`.
- В рабочих экранах доступны отдельные действия скачивания: Word, PDF, фото.

## БП-1: Создание новой заявки

### Описание
Фасилитатор регистрирует бенефициара и создает грантовую заявку.

### Предусловия
- Бенефициар имеет статус `certified` в базе
- Нет блокирующих дублей по ИНН или телефону

### Последовательность действий

```mermaid
sequenceDiagram
    actor Ф as Фасилитатор
    participant UI as Интерфейс
    participant DB as База бенефициаров
    participant ST as State (заявки)

    Ф->>UI: Нажимает "Дархости нав / Новая заявка"
    UI->>UI: Показывает блок createFormBlock
    Ф->>UI: Вводит ФИО / ИНН / телефон в поиск
    UI->>DB: Поиск по mockDatabase + beneficiarySearchDatabase
    DB-->>UI: Результаты (список бенефициаров)
    
    alt Найден дубль по ИНН/телефону
        UI->>UI: Показывает предупреждение (красное)
        UI->>UI: Блокирует кнопку "Создать"
    else Дубль по ФИО (мягкий)
        UI->>UI: Показывает предупреждение (желтое)
        UI->>UI: Кнопка "Создать" доступна
    end

    Ф->>UI: Выбирает бенефициара из выпадающего списка
    UI->>UI: Отображает выбранного с инициалами и статусом
    
    alt certStatus != "certified"
        UI->>UI: Блокирует кнопку "Создать"
    end

    Ф->>UI: Нажимает "Создать заявку"
    UI->>ST: Создает заявку (status: draft)
    UI->>UI: Открывает модальное окно → таб "Фасилитатор"
    
    Note over Ф,ST: Заявка заполняется в модальном окне
    
    Ф->>UI: Заполняет сектор и сумму
    
    alt Сохранить как черновик
        Ф->>UI: Нажимает "Захира / Сохранить черновик"
        alt Данные бенефициара полные
            UI->>ST: app.status = \"draft\", addLog()
        else Данные бенефициара неполные
            UI->>ST: app.status = \"incomplete_data\", missingFields = [...]
            UI->>UI: Показывает оранжевый блок предупреждения
            UI->>UI: Поля с пропусками подсвечены красным
            UI->>UI: Кнопка «Отправить в ШИГ» заблокирована
        end
    else Отправить в ШИГ / КУГ
        alt Данные бенефициара неполные
            UI->>UI: Предупреждение (toast): «Маълумоти бенефициар нопурра аст!»
            UI->>UI: Отправка заблокирована
        else Данные полные
            Ф->>UI: Нажимает \"Ба ШИГ фиристодан / Отправить в КУГ\"
            UI->>ST: app.status = \"gmc_review\", addLog()
        end
    end
    
    UI->>UI: Закрывает модалку, перерисовывает карточки
```

### Где отображается
| Элемент | ID в DOM | Описание |
|---------|----------|----------|
| Кнопка "Новая заявка" | `toggleFormBtn` | Шапка страницы, правый угол |
| Блок создания | `createFormBlock` | Под шапкой, скрыт по умолчанию |
| Поиск бенефициара | `beneficiarySearch` | Текстовое поле в блоке создания |
| Выпадающий список | `searchDropdown` | Под полем поиска |
| Предупреждение о дубле | `beneficiary-duplicate-warning` | Под полем поиска |
| Кнопка создания | `submitApplicationBtn` | В блоке создания |

## БП-2: Оценка заявки в ШИГ / КУГ (GMC)

### Описание
Специалист ШИГ / КУГ проводит скоринг заявки по 15 критериям и 3 критериям соответствия.

### Диаграмма процесса

```mermaid
sequenceDiagram
    actor Г as Специалист ШИГ / КУГ
    participant UI as Интерфейс (таб ШИГ / КУГ)
    participant ST as State

    Г->>UI: Открывает заявку из дашборда
    UI->>UI: Открывает модалку → таб "ШИГ / КУГ"
    UI->>UI: Загружает форму оценки (loadGmcForm)
    
    Note over UI: 3 критерия соответствия (el1-el3): Да/Нет
    Note over UI: 15 критериев скоринга (q1-q15): 1-4 балла
    Note over UI: Макс. возможный балл: 60
    
    Г->>UI: Заполняет оценки
    UI->>UI: Автоматический пересчет суммы баллов
    
    alt Есть "Нет" в критериях соответствия
        UI->>UI: Блокируется "Рекомендовать" и "Доработка"
        UI->>UI: Доступно только "Отклонить"
    else Балл ≥ 45
        UI->>UI: Доступно: "Рекомендовать"
        UI->>UI: Заблокировано: "Доработка", "Отклонить"
    else Балл 30-44
        UI->>UI: Доступно: "На доработку"
        UI->>UI: Заблокировано: "Рекомендовать", "Отклонить"
    else Балл < 30
        UI->>UI: Доступно: "Отклонить"
        UI->>UI: Заблокировано: "Рекомендовать", "Доработка"
    end
    
    alt Рекомендовать (≥ 45)
        Г->>UI: Нажимает "Тавсия додан"
        UI->>ST: app.status = "gmc_preparation"
        UI->>ST: addLog("Тасдиқ шуд, барои омодасозии реестр гузашт")
    else На доработку (30-44)
        Г->>UI: Нажимает "Ба такмил"
        UI->>ST: app.revisionCount++
        alt revisionCount >= 3
            UI->>ST: app.status = "postponed"
            UI->>ST: addLog("Лимит доработок исчерпан. Отложено на 3 месяца")
        else revisionCount < 3
            UI->>ST: app.status = "fac_revision"
            UI->>ST: addLog("Возвращено на доработку Фасилитатору")
        end
    else Отклонить
        Г->>UI: Нажимает "Рад кардан"
        UI->>ST: app.status = "rejected"
        UI->>ST: addLog("Заявка отклонена")
    end
```

### Правила скоринга

```
┌───────────────────────────────────────────────────┐
│                  СКОРИНГ ШИГ                      │
├───────────────┬───────────────────────────────────┤
│  ≥ 45 баллов  │  → Рекомендовать к подготовке реестра │
│  30-44 балла  │  → На доработку Фасилитатору      │
│  < 30 баллов  │  → Отклонить                      │
│  "Нет" в el*  │  → Только отклонить               │
├───────────────┴───────────────────────────────────┤
│  Лимит доработок: 3 попытки                       │
│  При исчерпании → статус "postponed" (3 мес.)     │
└───────────────────────────────────────────────────┘
```

## БП-5: Утверждение Комитетом

### Описание
Комитет рассматривает реестр заявок и принимает решение по каждой.

```mermaid
sequenceDiagram
    actor К as Член Комитета
    participant UI as Интерфейс (список Комитета)
    participant ST as State

    К->>UI: Переходит в раздел "Кумита / Комитет"
    UI->>UI: Показывает входящие реестры (карточки)
    
    К->>UI: Открывает реестр
    UI->>UI: Модалка → таб "Рӯйхати Кумита"
    UI->>UI: Таблица заявок с dropdown решений

    loop Для каждой заявки в реестре
        К->>UI: Выбирает "Одобрить" или "Отклонить"
        alt Отклонить
            UI->>UI: Показывает inline-поле комментария
            К->>UI: Вводит причину (обязательно)
            Note over К,UI: Без причины утверждение списка блокируется
        end
    end

    Note over К,UI: ❗ Дата протокола обязательна.<br/>Без неё → warning toast: «Выберите дату»

    К->>UI: Нажимает "Тасдиқ кардан / Утвердить список"
    
    loop Для каждой заявки
        alt Одобрена
            UI->>ST: app.status = "approved"
            UI->>ST: app.protocolId = номер протокола
            UI->>ST: addLog("Грант утвержден")
            UI->>ST: generateMonitoringFor(app.id) → 4 визита
        else Отклонена
            UI->>ST: app.status = "postponed"
            UI->>ST: app.committeeReturnsCount++
            UI->>ST: app.lastCommitteeReturn = {protocolId, date, time, comment}
            UI->>ST: addLog("Отклонено Комитетом, заявка отложена на 3 месяца", comment)
        end
    end

    UI->>ST: Создается протокол в protocols[]
    UI->>UI: Переключает на "Тасдиқшуда / Одобренные"
```

### Структура утверждения

```
┌─────────────────────────────────────────────────┐
│            ЗАСЕДАНИЕ КОМИТЕТА                    │
│                                                 │
│  Входящий реестр: РЕЕСТР-GMS-1001               │
│  Номер протокола: ПР-XXXX (авто)               │
│  Дата: выбирается                               │
│                                                 │
│  ┌───┬──────────────┬─────────┬──────────┐      │
│  │ # │ Заявитель    │ Сумма   │ Решение  │      │
│  ├───┼──────────────┼─────────┼──────────┤      │
│  │ 1 │ Тоиров Б.    │ 10 500  │ ✅ Одобр.│      │
│  │ 2 │ Каримов Р.   │ 15 000  │ ❌ Откл. │      │
│  └───┴──────────────┴─────────┴──────────┘      │
│                                                 │
│  [📥 Word] [📄 PDF] [🖼 Фото] [Тасдиқ кардан / Утвердить] │
└─────────────────────────────────────────────────┘
```

## БП-6A: Формирование черновика договора в системе

### Описание
После перехода заявки в `approved` Фасилитатор заполняет демо-форму договора в системе, сохраняет черновик и формирует документ для печати/PDF.

```mermaid
sequenceDiagram
    actor Ф as Фасилитатор
    participant UI as Интерфейс (full-width блок договора)
    participant ST as State

    Ф->>UI: Открывает approved-заявку
    UI->>UI: Нажимает "Создать договор"
    UI->>UI: Раскрывается аккордеон формы
    UI->>ST: merge(defaults из app/db + existing draft)
    Ф->>UI: Заполняет ручные поля
    UI->>UI: Нормализует номер в формат Ш-******-ДДММГГ
    Ф->>UI: Нажимает "Сохранить черновик"
    UI->>ST: app.grantContractDraft = { fields, updatedAt, updatedByRole }
    Ф->>UI: Открывает предпросмотр / печать / экспорт PDF
    Ф->>UI: При необходимости нажимает нижнюю кнопку "Бекор кардан / Отмена"
```

Ключевые правила:
- Раздел доступен только Фасилитатору и только в `approved`.
- Порядок полей в форме соответствует структуре договора.
- Сумма гранта подтягивается автоматически из заявки.
- Для предпросмотра/печати/PDF применяется строгая проверка обязательных полей.

## БП-6: Подписанный договор после одобрения

### Описание
После перехода заявки в `approved` Фасилитатор прикрепляет подписанный скан договора о гранте.

```mermaid
sequenceDiagram
    actor Ф as Фасилитатор
    participant UI as Интерфейс (таб История/Финальные документы)
    participant ST as State

    Ф->>UI: Открывает approved-заявку
    UI->>UI: Видит блок "Подписанный договор"
    Ф->>UI: Выбирает файл (PDF/JPG/JPEG/PNG)
    Ф->>UI: Нажимает "Загрузить"
    UI->>ST: app.grantAgreement = {fileName, uploadedAt, uploadedByRole, note}
    UI->>ST: addLog("Загружен подписанный договор")
    UI->>UI: Показывает метаданные + кнопку скачивания
```

Ключевые правила:
- Загрузка доступна только для `approved`.
- Загрузка доступна только Фасилитатору.
- Каждое обновление договора фиксируется в истории.

## БП-7: Мониторинг выданного гранта

### Описание
После утверждения автоматически создается график из 4 мониторинговых визитов.

```mermaid
sequenceDiagram
    actor Ф as Фасилитатор
    participant UI as Интерфейс (таб Мониторинг)
    participant ST as State.monitoring

    Note over ST: При утверждении автоматически создается:<br/>Визит 1 (+30 дней) — active<br/>Визит 2 (+90 дней) — pending<br/>Визит 3 (+180 дней) — pending<br/>Визит 4 (+360 дней) — pending

    Ф->>UI: Открывает одобренную заявку → таб Мониторинг
    UI->>UI: renderMonitoringList()
    UI->>UI: Визит 1 — желтая карточка (active)
    UI->>UI: Визиты 2-4 — серые (pending)

    Ф->>UI: Нажимает "Пур кардан / Оформить" на визите 1
    UI->>UI: Раскрывает форму визита

    Note over UI: Форма визита содержит:
    Note over UI: 1. Состояние оборудования (в наличии/не используется/продано)
    Note over UI: 2. Состояние бизнеса (активен/приостановлен/закрыт)
    Note over UI: 3. Средний месячный доход
    Note over UI: 4. Экологические стандарты (да/нет)
    Note over UI: 5. Фотографии (мин. 2)
    Note over UI: 6. Примечание
    Note over UI: ❗ Поля 1-4 обязательны.<br/>Без них → warning toast: 'Лутфан ҳамаи майдонҳои ҳатмиро пур кунед!'

    Ф->>UI: Заполняет все поля
    
    alt Оборудование "Продано"
        UI->>UI: Показывает предупреждение
        UI->>Ф: "Огоҳинома ба Админ!" (алерт)
    end

    Ф->>UI: Нажимает "Захира / Сохранить"
    UI->>ST: visit.status = "completed"
    UI->>ST: Следующий визит → status = "active"
    UI->>UI: Перерисовка списка визитов
```

### Жизненный цикл визитов

```mermaid
stateDiagram-v2
    [*] --> active_1: Грант утверждён (+30 дней)
    active_1 --> completed_1: Фасилитатор заполнил
    completed_1 --> active_2: Автоматически (+90 дней)
    active_2 --> completed_2: Фасилитатор заполнил
    completed_2 --> active_3: Автоматически (+180 дней)
    active_3 --> completed_3: Фасилитатор заполнил
    completed_3 --> active_4: Автоматически (+360 дней)
    active_4 --> completed_4: Фасилитатор заполнил
    completed_4 --> [*]: Мониторинг завершен
```

## БП-8: Доработка заявки (revision cycle)

### Описание
Заявка возвращается на доработку из ШИГ / КУГ к Фасилитатору. При отклонении Комитетом заявка переводится в `postponed` (3 месяца). Максимум 3 попытки доработки.

```mermaid
flowchart TD
    A[Заявка в ШИГ / КУГ<br/>gmc_review] -->|Балл 30-44| B[Возврат Фасилитатору<br/>fac_revision]
    B --> C{revisionCount < 3?}
    C -->|Да| D[Фасилитатор дорабатывает]
    D --> E[Повторная отправка<br/>gmc_review]
    E --> A
    C -->|Нет, 3/3| F[Система<br/>postponed<br/>Блокировка 3 месяца]
    F -->|Срок истек| G[Готова к разблокировке<br/>unlock-ready]
    G -->|Ручная разблокировка| D

    H[Заявка в Комитете<br/>com_review] -->|Отклонение| I[postponed<br/>3 месяца]
    I -->|Срок истек + ручная разблокировка| D

    style B fill:#fff3cd
    style F fill:#f8d7da
```

### Обязательные условия при возвратах

| Возврат | Обязательное поле | Уведомление при нарушении |
|---------|-------------------|---------------------|
| ШИГ / КУГ → Фасилитатор (`saveGmcDecision(rev)`) | Комментарий | `'Лутфан эзоҳи бозгардониданро нависед!'` |
| Комитет → `postponed` | Комментарий | фиксируется в возврате Комитета |
| ШИГ / КУГ: лимит 3/3 → postponed | — (автоматически) | `'Лимити такмил (3/3) ба итмом расид. Дархост ба таъхир гузошта шуд.'` |

### Счетчик доработок

```
Попытка 1/3 → fac_revision (можно доработать)
Попытка 2/3 → fac_revision (можно доработать)
Попытка 3/3 → postponed (блокировка на 3 месяца)
              → после истечения срока: unlock-ready (готова к разблокировке)
              → ручная разблокировка Фасилитатором: fac_revision, reactivated = true, revisionCount = 0
```

### Документный цикл при доработках

```mermaid
sequenceDiagram
    participant F as Фасилитатор
    participant G as ШИГ / КУГ
    participant C as Комитет
    participant ST as State.documents

    F->>ST: Первичная подача: Word V1 + basePdf + 4 basePhotos
    G->>C: Передача в составе реестра (текущий Word)
    C->>F: Отклонение, заявка уходит в postponed
    F->>ST: После разблокировки registerWordVersion() -> Word V2
    F->>G: Повторная отправка по процессу с новой Word версией

    Note over ST: basePdf и basePhotos остаются неизменными
```

## БП-9: Экспорт протокола

### Описание
Утверждённый протокол можно экспортировать в CSV.

```mermaid
sequenceDiagram
    actor К as Член Комитета
    participant UI as Интерфейс
    participant Browser as Браузер

    К->>UI: Открывает утверждённый протокол
    UI->>UI: Показывает кнопку экспорта
    К->>UI: Нажимает "Содирот / Экспорт"
    UI->>UI: Формирует CSV с BOM (UTF-8)
    UI->>Browser: Blob → ObjectURL → download
    Browser->>К: Скачивается файл Список_ПР-XXXX_дата.csv
```

### Формат CSV

```
Давр;Ном;Насаб;Рақами инфиродӣ;Шаҳр;Ноҳия;Номгӯйи тиҷорат;Санаи пешниҳоди НС;Маблағи грант (сомонӣ);Номи нархгузор;Тасдиқ гардида
"СП-9001";"Бахтиёр";"Тоиров";"9876543210";"Хуҷанд";"Бобоҷон Ғафуров";"Савдо";"13.03.2026";"10500";"ШИГ / КУГ";"Ҳа"
```
