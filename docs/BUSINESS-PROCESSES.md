## БП-1: Создание новой заявки

### Описание
Фасилитатор регистрирует бенефициара и создает грантовую заявку.

### Предусловия
- Бенефициар имеет статус `certified` в базе
- Нет блокирующих дублей по ИНН или телефону

### Последовательность действий

```mermaid
sequenceDiagram
    actor Ф as Фасилитатор
    participant UI as Интерфейс
    participant DB as База бенефициаров
    participant ST as State (заявки)

    Ф->>UI: Нажимает "Дархости нав / Новая заявка"
    UI->>UI: Показывает блок createFormBlock
    Ф->>UI: Вводит ФИО / ИНН / телефон в поиск
    UI->>DB: Поиск по mockDatabase + beneficiarySearchDatabase
    DB-->>UI: Результаты (список бенефициаров)
    
    alt Найден дубль по ИНН/телефону
        UI->>UI: Показывает предупреждение (красное)
        UI->>UI: Блокирует кнопку "Создать"
    else Дубль по ФИО (мягкий)
        UI->>UI: Показывает предупреждение (желтое)
        UI->>UI: Кнопка "Создать" доступна
    end

    Ф->>UI: Выбирает бенефициара из выпадающего списка
    UI->>UI: Отображает выбранного с инициалами и статусом
    
    alt certStatus != "certified"
        UI->>UI: Блокирует кнопку "Создать"
    end

    Ф->>UI: Нажимает "Создать заявку"
    UI->>ST: Создает заявку (status: draft)
    UI->>UI: Открывает модальное окно → таб "Фасилитатор"
    
    Note over Ф,ST: Заявка заполняется в модальном окне
    
    Ф->>UI: Заполняет сектор и сумму
    
    alt Сохранить как черновик
        Ф->>UI: Нажимает "Захира / Сохранить черновик"
        UI->>ST: app.status = "draft", addLog()
    else Отправить в КУГ
        Ф->>UI: Нажимает "Ба ШИГ фиристодан / Отправить в КУГ"
        UI->>ST: app.status = "gmc_review", addLog()
    end
    
    UI->>UI: Закрывает модалку, перерисовывает карточки
```

### Где отображается
| Элемент | ID в DOM | Описание |
|---------|----------|----------|
| Кнопка "Новая заявка" | `toggleFormBtn` | Шапка страницы, правый угол |
| Блок создания | `createFormBlock` | Под шапкой, скрыт по умолчанию |
| Поиск бенефициара | `beneficiarySearch` | Текстовое поле в блоке создания |
| Выпадающий список | `searchDropdown` | Под полем поиска |
| Предупреждение о дубле | `beneficiary-duplicate-warning` | Под полем поиска |
| Кнопка создания | `submitApplicationBtn` | В блоке создания |

## БП-2: Оценка заявки в ШИГ/КУГ (GMC)

### Описание
Специалист ШИГ проводит скоринг заявки по 15 критериям и 3 критериям соответствия.

### Диаграмма процесса

```mermaid
sequenceDiagram
    actor Г as Специалист ШИГ
    participant UI as Интерфейс (таб ШИГ)
    participant ST as State

    Г->>UI: Открывает заявку из дашборда
    UI->>UI: Открывает модалку → таб "ШИГ"
    UI->>UI: Загружает форму оценки (loadGmcForm)
    
    Note over UI: 3 критерия соответствия (el1-el3): Да/Нет
    Note over UI: 15 критериев скоринга (q1-q15): 1-4 балла
    Note over UI: Макс. возможный балл: 60
    
    Г->>UI: Заполняет оценки
    UI->>UI: Автоматический пересчет суммы баллов
    
    alt Есть "Нет" в критериях соответствия
        UI->>UI: Блокируется "Рекомендовать" и "Доработка"
        UI->>UI: Доступно только "Отклонить"
    else Балл ≥ 45
        UI->>UI: Доступно: "Рекомендовать"
        UI->>UI: Заблокировано: "Доработка", "Отклонить"
    else Балл 30-44
        UI->>UI: Доступно: "На доработку"
        UI->>UI: Заблокировано: "Рекомендовать", "Отклонить"
    else Балл < 30
        UI->>UI: Доступно: "Отклонить"
        UI->>UI: Заблокировано: "Рекомендовать", "Доработка"
    end
    
    alt Рекомендовать (≥ 45)
        Г->>UI: Нажимает "Тавсия додан"
        UI->>ST: app.status = "piu_review"
        UI->>ST: addLog("Тасдиқ шуд, ба ГРП равон шуд")
    else На доработку (30-44)
        Г->>UI: Нажимает "Ба такмил"
        UI->>ST: app.revisionCount++
        alt revisionCount >= 3
            UI->>ST: app.status = "postponed"
            UI->>ST: addLog("Лимит доработок исчерпан. Отложено на 3 месяца")
        else revisionCount < 3
            UI->>ST: app.status = "fac_revision"
            UI->>ST: addLog("Возвращено на доработку Фасилитатору")
        end
    else Отклонить
        Г->>UI: Нажимает "Рад кардан"
        UI->>ST: app.status = "rejected"
        UI->>ST: addLog("Заявка отклонена")
    end
```

### Правила скоринга

```
┌───────────────────────────────────────────────────┐
│                  СКОРИНГ ШИГ                      │
├───────────────┬───────────────────────────────────┤
│  ≥ 45 баллов  │  → Рекомендовать в ГРП            │
│  30-44 балла  │  → На доработку Фасилитатору      │
│  < 30 баллов  │  → Отклонить                      │
│  "Нет" в el*  │  → Только отклонить               │
├───────────────┴───────────────────────────────────┤
│  Лимит доработок: 3 попытки                       │
│  При исчерпании → статус "postponed" (3 мес.)     │
└───────────────────────────────────────────────────┘
```

## БП-3: Проверка ГРП / PIU

### Описание
ГРП проводит социально-экологическую оценку заявки.

```mermaid
sequenceDiagram
    actor П as Специалист ГРП
    participant UI as Интерфейс (таб ГРП)
    participant ST as State

    П->>UI: Открывает заявку (openPiuFor)
    UI->>UI: Заголовок с именем и ID заявки
    UI->>UI: Показывает форму PIU-проверки

    П->>UI: Нажимает "Ворид кардан / Внести"
    UI->>UI: Раскрывает форму решения

    alt Одобрить
        П->>UI: Выбирает "Тасдиқ / Одобрить"
        UI->>UI: Зеленая подсветка кнопки
    else Вернуть на доработку
        П->>UI: Выбирает "Ба такмил / На доработку"
        UI->>UI: Оранжевая подсветка, появляется поле комментария
        П->>UI: Вводит комментарий (обязательно)
        Note over UI: Без комментария → alert:<br/>'Лутфан сабаби баргардониданро нависед!'
    end

    П->>UI: Сохраняет решение (savePiuCard)
    UI->>UI: Счетчик показывает 1/1

    П->>UI: Нажимает "Анҷом додан / Завершить"
    
    alt Есть возврат на доработку
        UI->>ST: app.status = "gmc_revision"
        UI->>ST: addLog("Возвращено с комментариями в КУГ")
    else Всё одобрено
        UI->>ST: app.status = "gmc_preparation"
        UI->>ST: addLog("Социально-экологическая оценка пройдена")
    end
```

## БП-4: Подготовка реестра для Комитета

### Описание
После прохождения ГРП-проверки, ШИГ готовит заявки в реестр для отправки в Комитет.

```mermaid
sequenceDiagram
    actor Г as Специалист ШИГ
    participant UI as Интерфейс
    participant ST as State

    Note over Г: Заявка в статусе gmc_preparation

    Г->>UI: Открывает заявку
    UI->>UI: Показывает блок "Верификация ГРП пройдена"
    UI->>UI: Кнопка "Ба реестр илова кардан / Добавить в реестр"

    Г->>UI: Нажимает кнопку
    UI->>ST: app.status = "gmc_ready_for_registry"
    UI->>ST: addLog("Заявка подготовлена для реестра Комитета")

    Note over Г: Заявка появляется в фильтре "Омода барои реестр"

    Г->>UI: Переходит в подфильтр "Готовы для реестра"
    UI->>UI: Показываются чекбоксы на карточках
    UI->>UI: Панель "Формирование реестра" (gmc-registry-bar)

    Г->>UI: Выбирает заявки чекбоксами
    UI->>UI: Счётчик выбранных обновляется
    Г->>UI: Нажимает "Создать реестр"
    UI->>UI: Открывает предпросмотр реестра (pane-gmc-registry-preview)
    
    Г->>UI: Подтверждает отправку в Комитет
    UI->>ST: Для каждой заявки: status = "com_review"
    UI->>ST: Создается запись в registryLists[]
```

### Схема формирования реестра

```mermaid
flowchart TD
    A[Заявки gmc_ready_for_registry] --> B{Выбраны чекбоксами?}
    B -->|Да| C[Предпросмотр реестра]
    C --> D[Подтверждение отправки]
    D --> E[Создание записи registryList]
    D --> F[Все выбранные → com_review]
    E --> G[Реестр появляется<br/>в блоке Комитета]
    B -->|Нет| H[Кнопка неактивна]
```

## БП-5: Утверждение Комитетом

### Описание
Комитет рассматривает реестр заявок и принимает решение по каждой.

```mermaid
sequenceDiagram
    actor К as Член Комитета
    participant UI as Интерфейс (список Комитета)
    participant ST as State

    К->>UI: Переходит в раздел "Кумита / Комитет"
    UI->>UI: Показывает входящие реестры (карточки)
    
    К->>UI: Открывает реестр
    UI->>UI: Модалка → таб "Рӯйхати Кумита"
    UI->>UI: Таблица заявок с dropdown решений

    loop Для каждой заявки в реестре
        К->>UI: Выбирает "Одобрить" или "Отклонить"
        alt Отклонить
            UI->>К: Запрашивает причину (window.prompt)
            Note over К,UI: Пустой ответ = отмена отклонения
        end
    end

    Note over К,UI: ❗ Дата протокола обязательна.<br/>Без неё → alert: «Выберите дату»

    К->>UI: Нажимает "Тасдиқ кардан / Утвердить список"
    
    loop Для каждой заявки
        alt Одобрена
            UI->>ST: app.status = "approved"
            UI->>ST: app.protocolId = номер протокола
            UI->>ST: addLog("Грант утвержден")
            UI->>ST: generateMonitoringFor(app.id) → 4 визита
        else Отклонена
            UI->>ST: app.status = "rejected"
            UI->>ST: addLog("Грант отклонен", comment)
        end
    end

    UI->>ST: Создается протокол в protocols[]
    UI->>UI: Переключает на "Тасдиқшуда / Одобренные"
```

### Структура утверждения

```
┌─────────────────────────────────────────────────┐
│            ЗАСЕДАНИЕ КОМИТЕТА                    │
│                                                 │
│  Входящий реестр: РЕЕСТР-GMS-1001               │
│  Номер протокола: ПР-XXXX (авто)               │
│  Дата: выбирается                               │
│                                                 │
│  ┌───┬──────────────┬─────────┬──────────┐      │
│  │ # │ Заявитель    │ Сумма   │ Решение  │      │
│  ├───┼──────────────┼─────────┼──────────┤      │
│  │ 1 │ Тоиров Б.    │ 10 500  │ ✅ Одобр.│      │
│  │ 2 │ Каримов Р.   │ 15 000  │ ❌ Откл. │      │
│  └───┴──────────────┴─────────┴──────────┘      │
│                                                 │
│  [📥 Бизнес-план]  [Тасдиқ кардан / Утвердить]  │
└─────────────────────────────────────────────────┘
```

## БП-6: Мониторинг выданного гранта

### Описание
После утверждения автоматически создается график из 4 мониторинговых визитов.

```mermaid
sequenceDiagram
    actor Ф as Фасилитатор
    participant UI as Интерфейс (таб Мониторинг)
    participant ST as State.monitoring

    Note over ST: При утверждении автоматически создается:<br/>Визит 1 (+30 дней) — active<br/>Визит 2 (+90 дней) — pending<br/>Визит 3 (+180 дней) — pending<br/>Визит 4 (+360 дней) — pending

    Ф->>UI: Открывает одобренную заявку → таб Мониторинг
    UI->>UI: renderMonitoringList()
    UI->>UI: Визит 1 — желтая карточка (active)
    UI->>UI: Визиты 2-4 — серые (pending)

    Ф->>UI: Нажимает "Пур кардан / Оформить" на визите 1
    UI->>UI: Раскрывает форму визита

    Note over UI: Форма визита содержит:
    Note over UI: 1. Состояние оборудования (в наличии/не используется/продано)
    Note over UI: 2. Состояние бизнеса (активен/приостановлен/закрыт)
    Note over UI: 3. Средний месячный доход
    Note over UI: 4. Экологические стандарты (да/нет)
    Note over UI: 5. Фотографии (мин. 2)
    Note over UI: 6. Примечание
    Note over UI: ❗ Поля 1-4 обязательны.<br/>Без них → alert: 'Лутфан ҳамаи майдонҳои ҳатмиро пур кунед!'

    Ф->>UI: Заполняет все поля
    
    alt Оборудование "Продано"
        UI->>UI: Показывает предупреждение
        UI->>Ф: "Огоҳинома ба Админ!" (алерт)
    end

    Ф->>UI: Нажимает "Захира / Сохранить"
    UI->>ST: visit.status = "completed"
    UI->>ST: Следующий визит → status = "active"
    UI->>UI: Перерисовка списка визитов
```

### Жизненный цикл визитов

```mermaid
stateDiagram-v2
    [*] --> active_1: Грант утверждён (+30 дней)
    active_1 --> completed_1: Фасилитатор заполнил
    completed_1 --> active_2: Автоматически (+90 дней)
    active_2 --> completed_2: Фасилитатор заполнил
    completed_2 --> active_3: Автоматически (+180 дней)
    active_3 --> completed_3: Фасилитатор заполнил
    completed_3 --> active_4: Автоматически (+360 дней)
    active_4 --> completed_4: Фасилитатор заполнил
    completed_4 --> [*]: Мониторинг завершен
```

## БП-7: Доработка заявки (revision cycle)

### Описание
Заявка может быть возвращена на доработку из ШИГ или ГРП. Максимум 3 попытки.

```mermaid
flowchart TD
    A[Заявка в ШИГ<br/>gmc_review] -->|Балл 30-44| B[Возврат Фасилитатору<br/>fac_revision]
    B --> C{revisionCount < 3?}
    C -->|Да| D[Фасилитатор дорабатывает]
    D --> E[Повторная отправка<br/>gmc_review]
    E --> A
    C -->|Нет, 3/3| F[Автоматически<br/>postponed<br/>Пауза 3 месяца]
    F -->|Через 3 мес.| G[Реактивация<br/>fac_revision<br/>reactivated: true]
    G --> D

    H[Заявка в ГРП<br/>piu_review] -->|Возврат| I[Возврат в ШИГ<br/>gmc_revision]
    I --> J{ШИГ исправил?}
    J -->|Отправить в ГРП| H
    J -->|Вернуть Фасилитатору| B

    style B fill:#fff3cd
    style F fill:#f8d7da
```

### Обязательные условия при возвратах

| Возврат | Обязательное поле | Alert при нарушении |
|---------|-------------------|---------------------|
| ШИГ → Фасилитатор (`sendGmcToFacilitator`) | Комментарий | `'Лутфан эзоҳи бозгардониданро нависед!'` |
| ГРП → ШИГ (resubmit в `savePiuCard`) | Комментарий | `'Лутфан сабаби баргардониданро нависед!'` |
| ШИГ: лимит 3/3 → postponed | — (автоматически) | `'Лимити такмил (3/3) ба итмом расид. Дархост ба таъхир гузошта шуд.'` |

### Счетчик доработок

```
Попытка 1/3 → fac_revision (можно доработать)
Попытка 2/3 → fac_revision (можно доработать)
Попытка 3/3 → postponed (пауза 3 месяца)
              → через 3 мес. → reactivated, revisionCount = 0
```

## БП-8: Экспорт протокола

### Описание
Утверждённый протокол можно экспортировать в CSV.

```mermaid
sequenceDiagram
    actor К as Член Комитета
    participant UI as Интерфейс
    participant Browser as Браузер

    К->>UI: Открывает утверждённый протокол
    UI->>UI: Показывает кнопку экспорта
    К->>UI: Нажимает "Содирот / Экспорт"
    UI->>UI: Формирует CSV с BOM (UTF-8)
    UI->>Browser: Blob → ObjectURL → download
    Browser->>К: Скачивается файл Список_ПР-XXXX_дата.csv
```

### Формат CSV

```
ID;Аризадиҳанда (Заявитель);Бахш (Сектор);Маблағ (Сумма);Қарор (Решение);Бизнес-план
10040;"Тоиров Бахтиёр";"Савдо / Торговля";10500;"Тасдиқ / Одобрено";"Бизнес_план_10040.txt"
```
