# Пользовательские инструкции / User Instructions

## Актуализация от 14.03.2026

- В форме договора добавлена нижняя кнопка `Бекор кардан / Отмена` рядом с действиями сохранения/предпросмотра.
- Системные предупреждения и подтверждения показываются через `AppNotify` (toast/confirm), а не через прямые браузерные pop-up.
- В предупреждении о неполных данных бенефициара отсутствующие поля отображаются локализованными названиями (TJ/RU).
- Формулировка предупреждения о неполных данных упрощена: удалён повтор `Нопурра / Неполное`, список полей идёт сразу после двоеточия.
- В блоке неполных данных унифицирована подпись списка полей (`Майдонҳо / Поля`) и уточнена формулировка о невозможности отправки в ШИГ.
- Унифицирован тон уведомлений: `Лутфан... / Пожалуйста...` используется для выбора/заполнения, `... нишон диҳед / Укажите...` — для обязательной причины/комментария.
- Внешний вид toast-уведомлений обновлён в мягком стиле: увеличенные скругления, больше воздушных отступов, белый фон и синяя типографика.
- В уведомлениях по маршрутам заявки добавлены точные указания направления: кому передана заявка и какой следующий статус установлен (формат `Маршрут: роль -> роль, статус`).
- Дополнительно покрыты fallback-сценарии: 3-е неодобрение (заявка остается в `postponed`), ручная разблокировка (`postponed -> fac_revision`) и одиночные решения Комитета (`approved`/`gmc_revision`).
- Все статусные уведомления унифицированы по структуре: `Что произошло` -> `Маршрут` -> `Следующий статус`.
- Из маршрутных уведомлений убраны англоязычные коды статусов (`gmc_review`, `piu_review` и т.д.): пользователю показываются понятные названия этапов на русском.
- Все системные уведомления `AppNotify` приведены к единому визуальному стилю и показываются по центру экрана.
- Для маршрутных уведомлений сохранена усиленная структура: отдельные строки `Что произошло`, `Маршрут`, `Следующий статус` для быстрого чтения оператором.
- В разделе `По статусам` добавлен отдельный точный фильтр `Полностью завершенные`: туда попадают только заявки в `approved`, по которым уже загружен подписанный договор.
- В карточках и таблице такие заявки помечаются явно: главный бейдж `Полностью завершена`, отметка `Договор загружен` и дата закрытия по времени загрузки договора.
- В главном меню добавлен отдельный режим `Молия / Финансы`: это тот же контур одобренных Комитетом заявок, но строго с условием прикрепленного подписанного договора.
- Над списком полностью завершенных заявок показывается отдельная summary-плашка с правилом отбора и количеством записей.
- Для финансового отдела доступна выгрузка `CSV`-выписки по полностью завершенным заявкам (ID, ФИО, сектор, сумма, протокол, дата/файл договора, кто загрузил).
- В опроснике ШИГ/КУГ включена строгая проверка обязательных пунктов: пока не отмечены все критерии `el1-el3` и оценки `q1-q15`, кнопки выбора решения (`Рекомендовать / На доработку / Отклонить`) заблокированы; пропуски подсвечиваются красным.
- Для пакета документов Фасилитатора Word-файл сделан опциональным: заявка может быть отправлена без Word, при этом факт загрузки Word (если он был загружен) сохраняется и версионируется.
- В верхнем меню выровнена геометрия кнопок фильтров: пункт `По статусам` больше не смещается вниз относительно соседних пунктов.
- Верхнее меню оптимизировано по ширине: пункт `По статусам` расширен (без обрезки текста), а вкладка `Молия / Финансы` перенесена в самый конец меню и сокращена по подписи.
- Для узкой ширины экрана включён компактный режим верхнего меню: у неактивных пунктов скрывается вторая строка подписи, активный пункт остаётся двухстрочным для читаемости.
- В модальном окне группового рассмотрения Комитета в списке заявок клик по ФИО заявителя открывает его полную карточку заявки в режиме просмотра.
- В карточке, открытой из группового списка Комитета, добавлена кнопка `Назад к списку`, чтобы вернуться в тот же входящий реестр без закрытия модального окна.
- Для карточки, открытой из группового списка Комитета, кнопка закрытия `X` работает так же мягко: возвращает в список, а не закрывает модальное окно целиком.
- Во вкладке ШИГ/КУГ добавлено отдельное поле `ФИО оператора ШИГ / КУГ`: имя сотрудника сохраняется в заявке и пишется в историю действий вместо обезличенного `ШИГ / КУГ`.
- Поле `ФИО оператора ШИГ / КУГ` сделано обязательным для действий по кнопкам ШИГ/КУГ: без заполнения поле решения и сохранение/маршрутизация блокируются.
- FAQ обновляется по шагам для пользователей с низкой цифровой грамотностью: упрощена верхняя панель (основные действия + блок `Дополнительно`), добавлен быстрый вход `Я не понимаю, что делать`, увеличены размеры кликабельных элементов и читаемость текста.
- В FAQ внедрен приоритетный порядок для новичка: после общих вопросов сразу идут `Типовые ошибки`, затем разделы по ролям; порядок синхронизирован в desktop/mobile оглавлении.
- Ключевые ответы FAQ переведены в единый шаблон: `Что делать` → `Где нажать` → `Результат` (полное покрытие: ошибки, Фасилитатор, ШИГ/КУГ, ГРП, Комитет, Мониторинг).
- FAQ синхронизирован с кодом и регламентами: уточнены рабочие статусы по ролям, причины сообщения `Это не зона работы роли`, условия отправки в ШИГ (включая сессии/дубли/полноту данных), поведение `postponed` и сценарий мониторинга при `equipment = sold`.
- В FAQ добавлен мини-блок `Маршруты статусов за 30 секунд`: один экран, простой язык без терминов, чтобы пользователь быстро понимал общий путь заявки.
- По запросу упрощён способ поиска информации в FAQ: удалены верхние хеш-теги, добавлен блок выбора роли `что делать с заявкой` с прямыми переходами к пошаговым инструкциям для каждого пользователя.
- В FAQ активированы переходы по всем ключевым ссылкам роли/помощи: ссылка не только прокручивает к блоку, но и включает нужный ролевой фильтр. Также добавлен расширенный блок `Расширенные инструкции для пользователей` с полным набором шагов для каждой роли.

## Назначение

Документ содержит детальные пошаговые сценарии работы по ролям: Фасилитатор, ШИГ / КУГ, ГТЛ / ГРП, Комитет.

## Как пользоваться этим документом

1. Найдите свою роль.
2. Выполняйте шаги строго по порядку.
3. После каждого сценария сверяйте результат в пункте `Ожидаемый результат`.
4. Если результат не совпал, используйте блок `Если не получается`.

## Общие правила

- Интерфейс двуязычный: таджикский + русский.
- Любое действие пишется в историю заявки (`auditLog`).
- После изменения данных карточки и таблицы обновляются автоматически.
- Для `approved` и `rejected` открытие доступно в режиме только просмотра из любой роли.
- Для режима `Финансовый отдел` показываются только полностью завершенные заявки: `approved` + загруженный подписанный договор.

## Фасилитатор

### Сценарий F1. Создание новой заявки

1. Нажмите `Дархости нав / Новая заявка`.
2. Введите минимум 3 символа ФИО, ИНН или телефона.
3. Выберите бенефициара из списка результатов.
4. Нажмите `Эҷод кардан / Создать заявку`.
5. Откройте вкладку Фасилитатора и проверьте автозаполненные поля.

Ожидаемый результат:
- Создана карточка со статусом `draft` или `incomplete_data`.

Если не получается:
- Проверьте, что у бенефициара `certStatus = certified`.
- Проверьте предупреждение о дубле по ИНН/телефону.

### Сценарий F2. Сохранение черновика

1. Заполните доступные поля формы Фасилитатора.
2. Нажмите `Захира / Сохранить черновик`.
3. Убедитесь, что запись добавилась в историю.

Ожидаемый результат:
- Данные сохранены.
- Если профиль бенефициара неполный, статус становится `incomplete_data`.

### Сценарий F3. Первичная отправка в ШИГ / КУГ

1. Проверьте, что заполнены поля `Сектор` и `Сумма`.
2. Загрузите пакет документов:
3. PDF бизнес-план.
4. Ровно 4 фото.
5. Word бизнес-план — по необходимости (опционально).
6. Нажмите `Ба ШИГ фиристодан / Отправить в КУГ`.

Ожидаемый результат:
- Статус меняется на `gmc_review`.
- В карточке отображается `Current Word Version: V1`.

Если не получается:
- Проверьте, что нет блокировки по дублю.
- Проверьте полноту профиля бенефициара (9 обязательных полей).

### Сценарий F4. Доработка после возврата (`fac_revision`)

1. Откройте заявку со статусом `fac_revision`.
2. Проверьте комментарий причины возврата в истории.
3. В блоке документов загрузите только новую Word-версию.
4. Нажмите повторную отправку в ШИГ / КУГ.

Ожидаемый результат:
- Статус снова `gmc_review`.
- Версия Word увеличилась (`V2`, `V3` и т.д.).

Важно:
- PDF и фото в этом сценарии не перезаписываются.

### Сценарий F5. Разблокировка после `postponed`

1. Перейдите в фильтр отложенных заявок.
2. Найдите заявку с признаком готовности к разблокировке (`unlock-ready`).
3. Нажмите `Разблокировать` в карточке, таблице или в колокольчике уведомлений.

Ожидаемый результат:
- Заявка переходит в `fac_revision`.
- `revisionCount` сбрасывается и заявка снова доступна для доработки.

Важно:
- Автоматической разблокировки нет.

### Сценарий F6. Формирование текста договора (`approved`)

1. Откройте одобренную заявку (`approved`).
2. Во вкладке истории/финальных документов перейдите к блоку `Договор с грантополучателем`.
3. Нажмите `Создать договор` или `Редактировать договор`.
4. Заполняйте поля по секциям сверху вниз:
5. `Реквизиты гранта`.
6. `Стороны`.
7. `Контакты`.
8. `Банковские данные`.
9. `Подписи`.
10. Проверьте номер договора: он генерируется системой автоматически в формате `Ш-******-ДДММГГ`.
11. Нажмите `Сохранить черновик`.
12. Нажмите `Предпросмотр`.
13. При необходимости выберите `Печать` или `Экспорт PDF`.
14. Для быстрого закрытия формы внизу нажмите `Бекор кардан / Отмена`.

Ожидаемый результат:
- Черновик договора сохранен.
- Документ открывается в окне предпросмотра и готов к печати/PDF.

Если не получается:
- Проверьте обязательные поля (подсветка красным).
- Проверьте, что номер не изменялся вручную (поле системное, только для чтения).

### Сценарий F7. Загрузка подписанного договора (`approved`)

1. В той же заявке откройте блок `Подписанный договор`.
2. Нажмите `Выбрать файл`.
3. Выберите PDF/JPG/JPEG/PNG до 10MB.
4. Добавьте комментарий (по необходимости).
5. Нажмите `Загрузить`.
6. Проверьте появление метаданных файла.
7. Нажмите `Скачать` для проверки выдачи файла.

Ожидаемый результат:
- Файл загружен и доступен для скачивания.
- В истории появилась запись о загрузке/обновлении договора.
- После загрузки подписанного договора заявка становится видна в точном фильтре `Полностью завершенные`.

## ШИГ / КУГ

### Сценарий G1. Первичная оценка заявки (`gmc_review`)

1. Откройте заявку в статусе `gmc_review`.
2. Обязательно укажите ФИО сотрудника в поле `ФИО оператора ШИГ / КУГ`.
3. Заполните 3 критерия соответствия (`yes/no`).
4. Заполните 15 критериев скоринга.
5. Добавьте комментарий.
6. Выберите решение:
7. `Одобрить`.
8. `На доработку`.
9. `Отклонить`.

Ожидаемый результат:
- При `Одобрить` заявка переходит в `piu_review`.
- При `На доработку` заявка возвращается Фасилитатору.
- При `Отклонить` заявка закрывается как `rejected`.

Важно:
- Без заполненного поля `ФИО оператора ШИГ / КУГ` кнопки решений и сохранение решения недоступны.
- Если хотя бы один пункт опросника не отмечен, сохранение блокируется.
- До 100% заполнения опросника выбор решения недоступен (кнопки disabled).
- Система прокручивает к первой ошибке и показывает пропущенные строки красным.

### Сценарий G2. Повторная обработка возврата (`gmc_revision`)

1. Откройте заявку в `gmc_revision`.
2. Проверьте/уточните поле `ФИО оператора ШИГ / КУГ`.
3. Изучите комментарий возврата (ГРП или Комитет).
4. Загрузите новую Word-версию.
5. Выберите маршрут:
6. Вернуть в ГРП.
7. Вернуть Фасилитатору с комментарием.

Ожидаемый результат:
- Статус меняется согласно выбранному маршруту.

### Сценарий G3. Подготовка и отправка в Комитет

1. Откройте заявки в `gmc_preparation`.
2. Проверьте/уточните поле `ФИО оператора ШИГ / КУГ`.
3. Переведите их в `gmc_ready_for_registry`.
4. Включите нужные заявки чекбоксами.
5. Нажмите создание реестра.
6. Отправьте реестр в Комитет.

Ожидаемый результат:
- Заявки переходят в `com_review`.

## ГТЛ / ГРП

### Сценарий P1. Проверка заявки (`piu_review`)

1. Откройте заявку в `piu_review`.
2. Проведите проверку шага(ов) и заполните решение.
3. Если возвращаете, обязательно внесите причину.
4. Нажмите завершение проверки.

Ожидаемый результат:
- При одобрении: `gmc_preparation`.
- При возврате: `gmc_revision`.

## Комитет

### Сценарий C1. Рассмотрение реестра и выпуск протокола

1. Откройте входящий реестр (`committee`).
2. Установите дату протокола.
3. Для каждой заявки выберите решение:
4. `Одобрить`.
5. `Отклонить`.
6. Для отклонения заполните комментарий.
7. Чтобы посмотреть полную карточку заявки, нажмите на ФИО заявителя в строке списка.
8. После просмотра нажмите `Назад к списку` в карточке, чтобы вернуться в тот же реестр.
9. Нажмите утверждение списка.
10. При необходимости нажмите экспорт протокола.

Примечание:
- Если карточка была открыта из входящего списка Комитета, нажатие `X` в правом верхнем углу тоже возвращает в этот список.

Ожидаемый результат:
- По заявкам выставлены финальные статусы `approved` или `gmc_revision`.
- В системе создан протокол.

## Мониторинг (`approved`)

### Сценарий M1. Заполнение активного визита

1. Откройте одобренную заявку.
2. Перейдите на вкладку мониторинга.
3. В активном визите нажмите `Оформить`.
4. Заполните обязательные поля:
5. Состояние оборудования.
6. Состояние бизнеса.
7. Средний доход.
8. Эко-проверка.
9. Добавьте фото и примечание.
10. Нажмите `Сохранить`.

Ожидаемый результат:
- Текущий визит становится `completed`.
- Следующий визит становится `active`.

## Частые проблемы и быстрые проверки

### Не отправляется заявка из Фасилитатора

1. Проверьте `Сектор` и `Сумма`.
2. Проверьте комплект `PDF + 4 фото` (Word-файл опционален).
3. Проверьте предупреждение о неполных данных бенефициара.
4. Проверьте блокировку по дублям.

### Неактивна форма договора

1. Проверьте, что статус заявки `approved`.
2. Проверьте, что вы в роли Фасилитатора.
3. Нажмите кнопку раскрытия блока договора.

### Заявка не выходит из `postponed`

1. Убедитесь, что срок блокировки истек.
2. Выполните ручную разблокировку Фасилитатором.
