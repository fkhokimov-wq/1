# Регламент обновления документации

## Актуализация от 14.03.2026

- При описании пользовательских сообщений фиксировать использование `AppNotify` (toast/confirm) как основного механизма.
- Для предупреждений о неполных данных указывать только локализованные названия полей (TJ/RU), без технических английских ключей.

## Цель

Гарантировать, что пользовательские инструкции и технические документы всегда соответствуют текущему поведению системы.

## Обязательное правило

Любое изменение в UI, статусах, ролях, валидациях, полях формы, документах, фильтрах или мониторинге считается незавершенным, пока не обновлены соответствующие файлы в `docs/`.

## Что обновлять при изменениях

### 1) Изменения интерфейса и кнопок

Обновить:
- `docs/UI-GUIDE.md`
- `docs/USER-INSTRUCTIONS.md`

### 2) Изменения бизнес-логики и статусов

Обновить:
- `docs/WORKFLOW.md`
- `docs/BUSINESS-PROCESSES.md`
- `docs/ACCESS-CONTROL.md`

### 3) Изменения структуры данных и полей

Обновить:
- `docs/DATA-MODEL.md`
- `docs/STATISTICS-MONITORING-DATA.md` (если поле влияет на отчеты/фильтры)

### 4) Изменения аналитики/мониторинга/фильтров

Обновить:
- `docs/STATISTICS-MONITORING-DATA.md`
- `docs/UI-GUIDE.md`

### 5) Изменения, влияющие на старт проекта или навигацию по docs

Обновить:
- `README.md`

## Definition of Done для задачи

Задача считается завершенной только если выполнены все пункты:
- Код обновлен.
- Пользовательские шаги в `docs/USER-INSTRUCTIONS.md` актуальны.
- Технические документы по затронутым областям актуальны.
- В README есть ссылка на новый документ (если добавлен).
- В текстах нет устаревших форматов/терминов.

## Быстрый чек-лист перед завершением задачи

- Изменилась кнопка/форма/поток в UI?
- Изменилась валидация или обязательные поля?
- Изменилась роль, доступ, read-only поведение?
- Изменился статус или переход статусов?
- Изменились поля сущностей (Application/Visit/Agreement/ContractDraft)?
- Изменились фильтры или метрики мониторинга?
- Изменились тексты уведомлений/подтверждений (toast/confirm)?
- Не показываются ли пользователю технические ключи полей вместо локализованных лейблов?

Если хотя бы на один вопрос ответ Да, обновление docs обязательно.

## Рекомендация по структуре инструкций для пользователей

Для каждого сценария использовать единый шаблон:
- Предусловия
- Пошаговые действия
- Ожидаемый результат
- Если не получается

## Ответственный порядок обновления

1. Обновить пользовательские шаги (`USER-INSTRUCTIONS.md`).
2. Обновить технические детали (UI/Workflow/Data Model/Access).
3. Обновить статистику/мониторинг (если затронуто).
4. Обновить README (ссылки и краткие правила).
